## 1. 设计原则

### 1.0 以人为本（Human-Centered Design）

**总纲原则**：所有设计决策优先从用户的实际使用场景和身体条件出发，而不是从技术实现或界面美观出发。

**会议室用户行为特征**：
- 用户在会议室中站立操作、移动范围受限、注意力集中在讨论内容上
- 界面应该"退到背后"，让用户专注于内容本身，而非操作界面
- 减少用户的认知负担：工具随手可得，操作不需要学习，错误可以轻松撤销

**包容性设计**：
- 考虑不同身高、不同惯用手的用户：交互元素集中在舒适触控区，不强迫用户做出不自然的身体动作
- 底部工具栏确保无论高矮，用户都能以自然姿势触达所有功能，无需走动或弯腰

**即时响应**：
- 快速响应，即时反馈：每一个触摸操作都应在 100ms 内给出视觉反馈，让用户感知到系统在响应
- 系统状态变化须有明确的视觉、触觉或声音提示，消除用户的不确定感

### 1.1 大屏站立操作优先

**场景假设**：用户站立，距屏幕 0.8~1.5 米，以手指或触控笔操作。大屏尺寸 55"~86"，竖向触达范围有限（腰线至视线高度）。

**最小触控目标**：
- 工具栏图标触控热区：最小 64dp × 64dp（参考 Material 3 推荐的 48dp，大屏站立场景上调）
- 图标视觉尺寸：32dp～40dp，触控热区四周留 12dp+ 的缓冲间距
- 工具间距：相邻工具按钮之间间距不少于 8dp，避免误触

**1 米外可读**：
- 工具栏图标需清晰、语义明确，线条粗细不低于 2dp（物理像素对应大屏分辨率下仍可辨识）
- 状态信息（页码、协同人数）字号不小于 18sp
- 颜色色块尺寸不小于 36dp × 36dp

### 1.2 最小干扰画布

会议场景下画布是核心焦点，UI 组件应遵循"能收则收"原则：
- 工具栏固定在底部边缘，不占据中心画布区域
- 弹出面板采用浮层形式，不推开画布布局
- 颜色选择、线宽切换在工具栏内完成，无需弹窗
- 所有确认弹窗（清屏、新建）应小而精准，避免遮盖大面积画布

### 1.3 直接操作优于菜单层级

用户是非技术型会议参与者，操作应直觉化：
- 主要工具（笔、橡皮、图形、颜色）一次触达，不超过 1 层层级
- 避免下拉菜单和隐藏在长按/右键中的功能，大屏触控不友好
- 线宽三档、颜色 4 个快捷色全部常驻展示，不收进下拉选择器

### 1.4 操作可逆，容错优先

白板书写容易出现误操作（尤其多人协同时）：
- 撤销/重做必须在视觉最显眼区域，**仅通过工具栏按钮触发**（不支持手势，避免与多人书写时的多点触控冲突）
- 清屏、删除页面操作必须有二次确认
- 页面管理 Bottom Sheet 通过上移/下移箭头按钮排序，无拖拽手势，排序操作可撤销

### 1.5 协同状态实时可感知

多设备协同时，用户需要随时了解"谁在连接、谁在哪里写"：
- 协同状态（在线/离线）以持久指示器展示，不依赖用户主动查看
- 每台设备分配独立颜色，光标/彩圈跨设备可区分
- 协同中翻页锁定状态需明确视觉提示（被锁时不可交互的反馈）

### 1.6 横屏强制，PAD 与大屏布局一致

- 仅支持横屏（landscape）模式，不响应竖屏旋转
- 大屏与 PAD 使用相同布局框架：**底部横排工具栏**，通过 dp 自适应缩放处理密度差异
- PAD 上底部工具栏图标间距可适当收窄（因距离更近），但保持同一套 UI 结构，工具排列顺序与大屏完全一致

### 1.7 深色模式同等支持

会议室可能关灯使用：
- 画布背景白色在深色模式下改为浅灰（#F5F5F5 → #1E1E1E，或保留白色由用户选择）
- 工具栏背景在深色模式下充分对比，确保图标辨识度
- 4 种快捷色及扩展色板在深色背景下均需测试对比度

### 1.8 触控安全区约束（⚠️ 关键）

**物理约束背景**：
- 商用大屏（55"~86"）安装时屏幕底部距地约 50~70cm，顶部距地约 130~175cm
- 用户站立时手臂舒适上举约 160cm，勉强触达约 185cm
- 因此屏幕**顶部 30~40%** 区域属于高难度触控区——短个子用户基本无法触达，所有用户均无法精确点击

**设计强制要求**：
- **所有可交互 UI 元素（按钮、图标、开关、弹窗的可操作区域）必须位于屏幕高度的下方 65% 以内**
- 屏幕顶部 35% 为纯画布区（可书写，但不放置任何需精确点击的 UI）
- 弹出浮层、确认弹窗一律从底部向上弹出（Bottom Sheet 模式），禁止居中弹出或从顶部弹出

**底部横排工具栏对此约束的满足情况**：
- ✅ 底部横排方案天然满足触控安全区约束：所有工具集中在屏幕底部一行，严格位于下方 65% 区域内
- ✅ 相比左右分离方案，底部横排更彻底地解决了这一问题：不存在任何工具需要用户向上伸手操作
- ✅ 商用大屏底部距地约 50~70cm，正是最舒适的触控高度，用户无需弯腰即可轻松触达
- ✅ 无论用户站在画布哪个位置，向下伸手等距触达所有工具，无需走动
- ❌ 废弃顶部状态栏方案（协同状态、扫码按钮不放顶部）
- ✅ 协同入口 + 扫码分享统一在底部工具栏功能区
- ✅ 任何弹窗（二次确认、文件管理操作、协同面板、页面管理）均为 Bottom Sheet

### 1.9 固定画布，无缩放无平移

- 所有版本均不支持书写时缩放或平移画布
- 理由：
  - 大屏是多人共用视图，一人缩放会打乱所有人的视角
  - 缩放/平移手势（双指捏合/拖拽）与多人书写的多点触控存在冲突，无法可靠区分
  - 大屏物理尺寸（55"~86"）已提供足够书写空间，不需要通过缩放扩展区域
- 空间不足时通过**多页管理**解决，而非缩小画布视图
- PAD 协同时视图与大屏保持同步，不支持独立缩放
- **回顾模式**（仅查看时缩放）暂不做，列入后续版本评估

### 1.10 输入自适应（Input-Adaptive Interaction）

**核心理念**：书写与擦除的切换由设备和输入源自动决定，用户无需手动切换工具状态。不同使用场景下，"用什么写、用什么擦"的判断由系统完成，操作与人的自然直觉一致。

**设备模式自动识别**：App 启动时根据屏幕对角线自动选择模式：
- 对角线 **> 30"** → **大屏模式**（会议室壁挂大屏场景）
- 对角线 **≤ 30"** → **PAD 模式**（平板手持/桌面场景）
- 用户可在设置中手动覆盖自动识别结果

**大屏模式**（多人站立，手指操作）：
- 小面积触点（指尖接触面积小）→ 判定为**书写**
- 大面积触点（手掌/掌根接触面积大）→ 判定为**擦除**
- 各触点簇独立判断，多人同时操作互不干扰，A 的手掌不影响 B 的书写
- 工具栏橡皮擦按钮作为**显式擦除模式**，用于需要精确控制擦除范围的场景

**PAD 模式**（单人手持，笔+手指操作）：
- 触控笔触点 → 判定为**书写**
- 手指触点 → 判定为**擦除**
- 符合"拿笔写、手指擦"的书写直觉，无需切换工具状态

**设计价值**：两种模式下书写与擦除均无需手动切换，消除了多人协同时频繁点击橡皮擦按钮再切回笔刷的认知摩擦，让操作"隐形化"，用户专注于内容本身。

PaintBoard App

│   ├── 画布区域（主体，四边均为纯书写区域，最大化书写面积）

│   └── 底部工具栏（横排，从左到右六个分区）

**层级说明**：
- 主白板界面是唯一的"页面"，其他全部以**Bottom Sheet**形式从底部弹出，禁止居中弹窗（避免弹窗关闭按钮出现在屏幕上半部）
- 无页面跳转（无 Activity 切换），所有交互在主界面内完成，保证书写状态不丢失
- 文件管理采用**用户主动管理**模式：无自动保存、无历史列表；用户通过"更多"菜单中的"保存"/"打开"/"新建白板"操作管理 .pb 文件，.pb 文件可跨设备传递，任何装有本 App 的设备均可打开

**系统行为 — 崩溃恢复**（非用户可见的 UI 入口）：
- App 在后台每 2 分钟静默写入临时恢复文件（不触发任何 UI 提示）
- App 重启时若检测到未保存的临时恢复文件，弹出一次性提示："发现上次未保存的内容，是否恢复？"[恢复][放弃]
- 用户确认或拒绝后，不再重复提示，临时文件随之清除

**便利贴 v1.5 架构预留说明**：
- 便利贴功能（[便利贴 v1.5] 占位）引入**可移动对象层**，与笔画层（不可移动）分离渲染
- v1.0 架构须预留以下接口，避免后续大规模重构：
  - 数据模型层：支持"对象类型"字段区分笔画与可移动对象（便利贴、图片等）
  - 渲染层：支持双层 Canvas 或对象层叠加渲染（下层笔画 Canvas + 上层对象 Canvas）
  - 协同同步层：对象位置/内容的 CRDT 或 OT 冲突合并预留接口
- v1.0 不实现便利贴交互，仅在工具栏绘图工具区保留占位按钮（禁用态），提示用户"便利贴即将在 v1.5 推出"

**便利贴 v1.5 交互方案（已最终确认）**：
- 移除双击编辑、长按移动等手势操作（大屏多人场景下手势易误触且无直觉）
- 每张便利贴卡片右上角常驻三个图标按钮：
  - ✏️ **编辑**：点击后重新进入书写模式，可修改卡片内容
  - ✋ **移动**：点击后显示拖拽手柄，用户可拖动整张卡片至新位置
  - 🗑 **删除**：点击后弹出确认提示，确认后删除该便利贴
- 三个按钮始终可见（常驻），无需长按或双击激活，符合「直接操作优于菜单层级」原则（见 1.3）

## 3. 主界面布局方案

### 3.1 布局策略选型

针对大屏会议场景，对比四种工具栏位置方案：

| 方案 | 工具栏位置 | 优势 | 劣势 |
|------|-----------|------|------|
| 方案 A | 左侧竖排 | 书写手不遮挡画布；大屏左边缘易触达 | 需要纵向滚动（若工具多） |
| 方案 B | 底部横排（早期评估） | 符合手机使用习惯 | 早期评估时颜色色块横排过多，空间有限 |
| 方案 C | 左侧竖排 + 右侧缩略图 | 功能分区清晰，主体画布完整 | ❌ 左右两端相距 160cm+，用户需来回走动操作不同功能区，体验差 |
| **方案 D** | **底部横排（当前方案）** | **等距触达所有工具，无需走动；底部高度最舒适；画布四边均为纯书写区** | 横向宽度需合理规划分区 |

**选定方案 D（底部横排）**：所有工具和页面管理统一在底部横排。**无顶部状态栏，无侧边栏**。

选定理由（以人为本，等距触达）：
- 无论用户站在画布哪个位置，向下伸手等距触达所有工具，无需走动
- 方案 C 被否决：大屏尺寸下左右两端相距 160cm+，操作不同功能区需来回走动，违背「以人为本」原则
- 商用大屏底部距地约 50~70cm，是最舒适的触控高度
- 颜色简化为 4 个快捷色后，横排空间完全足够
- 画布四边均为纯书写区域，最大化书写面积

## 4. 工具栏设计

### 4.1 工具分组原则

底部横排工具栏分为六个横向分区，以 ‖ 分隔线区分：

```
区域 A — 绘图工具区    (互斥选择，当前激活态高亮)           [✏笔][🧹橡皮][⬡图形]
区域 B — 粗细区        (三档常驻，与激活工具联动)           [━细][─中][▬粗]
区域 C — 颜色区        (4 个快捷色 + 更多颜色按钮)         [●黑][●红][●蓝][●自定][🎨]
区域 D — 操作区        (撤销/重做)                        [↩][↪]
区域 E — 功能区        (协同/分享/更多)                    [🔗协同][📤分享][⋯更多]
区域 F — 页面区        (页码导航 + 新建页)                 [◀ 2/5 ▶][＋]
```

**宽度估算**（大屏 1920dp 横向空间）：
- 各区域工具按钮：64dp 热区 × 工具数量
- 区域间分隔线：8dp × 5 条 = 40dp
- 估算总占用：(3+3+5+2+3+2) × 64dp + 40dp ≈ 1,168dp，占 1920dp 的 61%，横向空间充裕

**底部工具栏高度**：
- 大屏：约 72dp（图标 40dp + 上下各 16dp 内边距）
- PAD：约 60dp（图标 32dp + 上下各 14dp 内边距）

### 4.2 常驻工具（工具栏直接展示）

橡皮擦工具激活时，向上弹出子面板浮层（与图形工具子面板行为一致），子面板内除橡皮大小切换外，还包含**清除当前页**功能：

```
│  ┌────────────────────────────────────────────────────────┐  │
│  │  橡皮大小：[● 细]  [● 中]  [● 大]                      │  │

└──────────────────────────────────────────────────────────────┘
```

**"清除当前页"交互规范**：
- **控件类型**：滑动解锁式（类似 iOS 关机滑块），轨道宽度填满子面板可用宽度
- **触发条件**：手指须主动从左端划过整个滑轨至右端方可触发，中途松手则自动回弹，不执行操作
- **无需弹窗确认**：滑动行为本身作为防误触机制，触发后直接清除，无二次确认弹窗
- **撤销支持**：清除操作纳入 Undo/Redo 栈，可通过工具栏撤销按钮恢复（与普通橡皮擦操作同等对待）

### 4.5 工具激活态视觉规范

- 未选中：图标线条色 `#757575`（灰色），背景透明
- 激活选中：图标线条色 `#FFFFFF`（白），背景 `#1976D2`（品牌蓝色圆角矩形），阴影轻微
- 禁用（如不可撤销）：图标线条色 `#BDBDBD`，不响应触控

### 4.6 颜色色块设计

```
  未选中态：     选中态：
  ┌──────┐       ┌──────┐
  │  ●   │       │  ●   │  ← 色块不变
  └──────┘       └──────┘
                 外圈白色选中环（3dp 宽）+ 蓝色外框（2dp）
```

- 色块直径：36dp
- 触控热区：64dp × 64dp（含间距缓冲）
- 黑色色块在深色工具栏背景下需要白色外描边（1dp）以保持可见

| ERR-03 | 扫码带走 WiFi 不同提示 | 手机不在同 WiFi 时的错误页说明 |
| ERR-04 | 导出失败（存储不足） | 系统错误处理 |

## 6. 设计文档规划

后续将输出以下具体设计文档，建议按优先级依次完成：

### 第一优先级（支撑开发启动）

#### `design-tokens.md` — 设计令牌规范
**内容**：颜色系统（亮色/深色 token）、字体排版（字号/行高/字重）、间距体系（4dp 基础网格）、圆角规范、阴影层级、动画时长与缓动曲线。
**用途**：开发基础样式配置，确保全局一致性。

#### `toolbar-component.md` — 工具栏组件规范
**内容**：底部横排工具栏完整尺寸标注（含六个分区的宽度分配）、每个按钮的 4 种状态（默认/悬停/激活/禁用）截图+标注、组件交互行为描述、大屏/PAD 尺寸差异对比表。
**用途**：开发底部工具栏 UI 的直接参考。

#### `canvas-interaction.md` — 画布交互规范
**内容**：书写手势定义（单指书写、多点触控优先级与冲突消解规则）、输入自适应识别逻辑（大屏模式：触点面积阈值判断书写/擦除；PAD 模式：触控笔 vs 手指判断书写/擦除；设备模式自动识别流程及手动覆盖入口）、橡皮擦光标跟随规范、图形绘制拖拽反馈、掌心擦除触发逻辑图、多点触控优先级。撤销/重做仅通过工具栏按钮触发，不绑定任何手势。
**用途**：解决开发中手势冲突的核心文档。

### 第二优先级（协同与分享功能）

#### `collaboration-ux.md` — 协同功能 UX 规范
**内容**：协同入口设计（底部工具栏功能区详细设计）、Host/PAD 角色下的 UI 状态差异、二维码展示浮层设计、在线用户颜色分配方案（8 色循环分配逻辑）、主持人锁定模式 UI、断线/重连状态提示设计、协同 UI 全状态流转图。
**用途**：协同功能 UI 实现的完整规范。

#### `share-qrcode-ux.md` — 扫码带走 UX 规范
**内容**：扫码带走二维码浮层设计（二维码尺寸计算、说明文字、有效期展示）、Web 端手机查看页面设计（响应式 HTML 布局线框图、图片卡片样式、PDF 下载按钮）、Token 过期错误页设计。
**用途**：Web 分享页的 UI 实现参考。

### 第三优先级（页面管理与文件管理）

#### `page-management-ux.md` — 页面管理 UX 规范
**内容**：页面管理 Bottom Sheet 详细设计（缩略图列表、当前页高亮、上移/下移箭头按钮排序交互说明——首页仅显示 ▼、末页仅显示 ▲、中间页两者均显示）、页码导航区交互设计（◀ 2/5 ▶ 点击触发逻辑）、新建页/删除页操作反馈、页面背景切换入口与预览、20 页上限状态。
**用途**：页面管理组件实现参考。

#### `file-management-export-ux.md` — 文件管理与导出 UX 规范
**内容**：保存 .pb 文件交互设计（命名输入框、系统存储选择器、成功反馈）、打开 .pb 文件交互设计（文件选择器、未保存内容三选一弹窗）、新建白板确认弹窗设计（三选一按钮布局）、导出进度反馈设计、崩溃恢复提示弹窗设计。
**用途**：文件管理和导出功能 UI 实现参考。

### 第四优先级（系统与边缘态）

#### `error-states.md` — 错误与空状态设计
**内容**：全部边缘场景的 UI 处理（见第 5 节 P2 列表）、Toast 提示规范（位置、持续时长、层级）、确认弹窗统一模板（图标/标题/正文/按钮排布）。
**用途**：确保异常路径有明确的用户反馈，防止开发遗漏。

#### `dark-mode-spec.md` — 深色模式规范
**内容**：亮色/深色模式下所有组件 token 映射、特殊处理项（黑色画笔在深色背景、白色预设色的展示）、大屏关灯场景下的亮度调节建议。
**用途**：深色模式适配参考。
